When this mode is entered, first load the `ralph` skill (the family map) — read `.claude/library/ralph/SKILL.md`, falling back to `~/.claude/library/ralph/SKILL.md` — then acknowledge it and **stop — wait for my instructions** before doing anything. Read the design — the spine `project/design/README.md`, and `project/design/INDEX.md` to enumerate the Decisions, pulling each `project/design/DNN.md` as needed — and decompose it into an ordered sequence of phases that the build loop will execute one at a time. This is iterative, not a single pass — draft the phases, then refine them with me: resize, reorder, split, and merge until each is right. Don't assume the plan is finished in one shot. Order is dependency order: each phase depends only on earlier ones. The plan says *what*; build supplies *how*. **Every phase must have a deterministic exit condition** — a mechanical pass/fail check (a green test/suite, an exit code, an exact match count) that is reproducible on identical repo state and whose passing state is actually **reachable**. A phase whose completion can only be judged subjectively ("states the truth", "documents adequately"), or whose check can never reach its passing state (e.g. a `grep` for a phrase the phase's own `project/` docs also contain, so it can never return empty), is **not yet a build phase** — reshape or split it until it can be mechanically checked, or it does not belong in the plan. This applies to structural/docs-only phases too: they earn a deterministic check, never a prose claim.

**Total coverage of the requirement denominator is mandatory.** Design's Verification ids (`R-XXXX-XXXX`) are the complete enumerated intent; the plan is not valid until **every** one of them is assigned to at least one build phase. Before you finalize, mechanically verify it: enumerate the full id set from design (`grep -hoE 'R-[A-Z0-9]{4}-[A-Z0-9]{4}' project/design/*.md | sort -u`, cross-checked against `project/design/INDEX.md`) and confirm each id appears in some `phase-NN.md`'s *Done when* (`grep -hoE 'R-[A-Z0-9]{4}-[A-Z0-9]{4}' project/plan/phase-*.md | sort -u`) — the two sets must be equal, with no id unassigned and no phased id absent from design. Every id lands in **exactly one** phase (no id split across phases, none duplicated). Because the plan is **strictly append-only** — a completed (`✅`) phase is never rewritten, so you can never go back and slot a forgotten id into it — this completeness must be right at authoring time: a requirement discovered or minted later can only be covered by a **new appended phase**, never by amending a finished one. So audit the id set to full equality before the plan is settled.

When we're settled, write the plan in the **split, addressable layout** below and report the paths.

## Why the plan is split

The plan owns **construction order and history** — the order the thing is built in and the record of what has been built. Unlike product and design (which are rewritten in place to stay authoritative for the current state), the plan is **append-only**: phases are added at the end and marked done as they land; completed phases are never rewritten or deleted, so the plan doubles as the construction history.

But an append-only plan grows without bound, and the build loop loads it on **every** turn just to find its next unit of work. So the plan is **physically split for addressability**: invariant rules in `project/plan/README.md`, a one-line-per-phase manifest in `project/plan/STATUS.md`, and one `project/plan/phase-NN.md` body file per phase. The loop greps the manifest for the next not-started phase and reads exactly that one phase file — never the whole history. Write all three pieces.

## `project/plan/README.md` — the invariant rules (static; never grows)

Holds only the rules below. It does **not** list phases or carry status, so it never grows with the project.

- **Title** — `# <name> — Plan`.
- **Authority header** — a paragraph beginning `**Authority: construction order and history.**` stating that this document **and the `project/plan/` directory it heads** own the build order and the record of what's built, that the plan is **append-only** (completed phases are never rewritten or deleted, so it doubles as history), and how to extend it: update product and design in place, then **append** a new phase — a new `phase-NN.md` plus a new `STATUS.md` line — never edit a finished phase except to flip its status marker. State the **coverage invariant** here too: the phases collectively assign **every** design Verification id (`R-XXXX-XXXX`) to exactly one phase, and because finished phases are frozen, any id minted or discovered after a phase is done can only be covered by a **newly appended** phase — never by amending a completed one.
- **One phase = one package = one build-turn context** — a short paragraph: each phase is a single coherent unit — almost always one package — scoped to that package's design Decisions and the *interfaces* (not internals) of the packages it depends on, and **sized so the gather→build→verify loop can carry it in one fresh `build`-turn context** and ideally finish it in a turn or two (with `verify` confirming on the next turn). Note the loop does *not* build a phase in one long accumulating context — `build` runs in a fresh context each turn and `verify` in another — so size to a single `build` turn, not an imagined single sitting. Sizing a phase as large as cleanly fits one `build` turn is good: it lets one turn cover several requirements, which means fewer cycles and less context churn. This is what keeps every phase the size of a small standalone tool no matter how large the project grows. (If a single package must split across phases to fit one context, say so.)
- **Done bar** — a paragraph stating a phase is **done** when every Verification item (the `R-XXXX-XXXX` ids) in the design Decisions it realizes — or the slice of those ids assigned to it — is covered by a clearly-named test and the suite is green; point at design's *Conventions* for what "green" concretely means and at each Decision's Verification list for what "covered" means; and state that every phase's acceptance bar must be expressed as **deterministic exit conditions** — mechanically-checkable, reproducible on identical repo state, and reachable — never a subjective prose judgment and never a self-referential or unsatisfiable check.
- **## Layout** — a short section describing the split so a reader knows where things live: `project/plan/STATUS.md` is the manifest and the **only** home of status markers; `project/plan/phase-NN.md` is one body file per phase (zero-padded; sub-phases keep their suffix, e.g. `phase-07a.md`); `project/plan/README.md` is these static rules. Restate the append-only rule for this layout: never rewrite or delete a `phase-NN.md`, never delete a `STATUS.md` line; the only build-time mutation is flipping one phase's `⬜ → ✅` in `STATUS.md`.

## `project/plan/STATUS.md` — the manifest (the only home of status markers)

- **Title** — `# <name> — Plan Status`.
- **Contract paragraph** — explain that this is the manifest: one line per phase in build order, the **only** place a phase's status marker lives; each phase line is a Markdown bullet beginning with `- Phase` and carries `✅` (done) or `⬜` (not started); the build loop finds its next work with `grep -nE '^- Phase .* ⬜' project/plan/STATUS.md | head -1`, reads only that phase's `project/plan/phase-NN.md`, and on completion flips that one marker. End the paragraph noting it deliberately carries **no bare status glyph**, so the anchored grep matches only phase lines.
- **The phase lines** — one per phase, each a Markdown bullet starting with `- Phase` and its zero-padded number, then the status marker, then `realizes <Decision ids>` (or `realizes —` for a pure structural phase), then `— <one cohesive objective>`. The bullet prefix makes each phase render as its own line in Markdown; keep them aligned and grep-able. This manifest is the single source of truth for status, so a phase file carries **no** marker of its own.

## `project/plan/phase-NN.md` — one body file per phase

One file per phase (zero-padded filename; sub-phases keep their suffix). Each holds:

- A header `# Phase N — <one cohesive objective>` — **no status token** in the header; status lives only in `STATUS.md`.
- A `*Realizes design Decision <n> (<short label>)[ and <m> (...)]. Depends on Phase <k>.*` line — name exactly which design Decisions this phase builds and which earlier phase(s) it needs. This link is what lets the build read only the relevant slice of design, not the whole doc.
- A short body: what gets built (the package/seam and its paths), stated as the observable end state, not an implementation recipe.
- **Done when:** the acceptance bar for this phase, stated as **deterministic exit conditions** — each a mechanically-checkable predicate (a green test/suite, an exit code, an exact match count) that is unambiguous, reproducible on identical repo state, and whose passing state is actually reachable: its design Verification ids covered by genuine tests, and the suite green. Never a subjective judgment ("states the truth", "documents adequately") and never a check that can't reach its passing state (e.g. a `grep` for a phrase the phase's own `project/` docs also contain, so it can never return empty). A **structural / docs-only phase** is held to the same bar — give it a deterministic check: the green build plus a *scoped* grep (excluding `project/` so it cannot match the workspace docs that quote the pattern) or a named integration smoke, not a prose claim.

Each phase must be **sized to a single fresh `build`-turn context** — small enough that one `build` invocation can read the brief, make the change, and leave the suite green (`verify` then confirms it on its own separate turn) — and ideally completable in a turn or two, not a long churn of thin partial increments. If a phase won't fit one package's worth of context, it's too big; split it before writing (and state the partial-Decision split in the affected phase files).
